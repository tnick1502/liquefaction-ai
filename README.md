# liquefaction-ai

Моделирование разжижения грунтов при циклическом нагружении с помощью трёх
физически-структурированных нейросетевых моделей **DPI-Flow**, **EVT-NeuralSSM** и их
объединения **DPI-EVT** с основной site-held-out оценкой. Задача формулируется как
**prefix-conditioned forecasting/continuation**: модель получает ранний наблюдаемый префикс
PPR и прогнозирует дальнейшую траекторию, риск и N_liq.
Проект строит воспроизводимый исследовательский пайплайн: от генерации физически
согласованных синтетических данных и адаптера реальных лабораторных опытов до отчётных
метрик, стресс-протоколов и интерактивной визуализации.

## Структура проекта

```
src/liquefaction_ai/        # пакет (вся логика и модели)
  config.py, constants.py   # конфигурация эксперимента и таксономия грунтов/нагружений
  data/                     # генерация популяции, грансостав, профиль грунта, сплиты, I/O
    grainsize.py            #   11 фракций, классификация PLAXIS и ГОСТ type_ground
    soil_profile.py         #   полный геотех. профиль грунта (физика + механика + CRR-параметры)
    synthetic.py            #   сборка популяции и интегрирование ODE повреждения/PPR/триггера
  physics/                  # аналитические модели CRR/PPR + физическая модель CRR(N)
    crr_physical.py         #   CRR(N)=β/N^(1−α) из физсвойств (перенос model.py)
  models/                   # нейросети: базовые (MLP/GRU/TCN) и DPI-Flow / EVT-NeuralSSM / DPI-EVT
  training/                 # функции потерь, цикл обучения, grid search, сохранение моделей
  evaluation/               # метрики качества, агрегаты, эксперименты
  viz/                      # matplotlib-тема (пастель Q1), построители графиков (с флагом save), кривые CRR/PPR

notebooks/
  1_data_analysis/          # 1.1 разведочный анализ, 1.2 анализ CRR-параметров, 1.3 сплиты (читают data/dataset)
  2_model_training/         # 2.1 базовые, 2.2 DPI-Flow, 2.3 EVT-NeuralSSM, 2.4 DPI-EVT (grid search → models/)
  3_evaluations/            # 3.1 метрики, 3.2 абляции/OOD, 3.3 разбор кейсов, 3.4–3.8 CI/значимость/публикация
  4_topology/               # 4.1 топология латента DPI-Flow, 4.2 ранний предвестник, 4.3 topo-регуляризация (extension)

data/    prepare_dataset.ipynb  # ЕДИНСТВЕННЫЙ ноутбук подготовки: sites/ → data/dataset (или синтетика)
data/    dataset/           # ЕДИНСТВЕННЫЙ артефакт данных (его читают все ноутбуки 1_*–4_*)
data/    sites/             # сырые пиклы циклических опытов (источник; нужен только prepare_dataset.ipynb)
models/  <модель>/          # обученные модели с гиперпараметрами и историей
results/ figs/              # сохранённые фигуры (.png + .pdf)
results/ tables/            # отчётные CSV-таблицы, генерируемые ноутбуками серии 3
```

Все результаты — это таблицы (`results/tables/*.csv`) и фигуры (`results/figs/*`), генерируемые
ноутбуками. Отдельного слоя PDF-отчётов в проекте нет: он исключён, чтобы не дублировать и
не рассинхронизироваться с CSV-таблицами — единственным источником истины.

## Пайплайн

1. **Подготовка данных (`data/prepare_dataset.ipynb`) и анализ (папка 1).** Источник выбирается
   через `LIQ_DATASET`: либо реальные опыты (сырые пиклы из `data/sites`, сборка в
   `liquefaction_ai.data.raw_loader`), либо синтетическая популяция грунтов с полным
   геотехническим набором — гранулометрический состав (11 фракций), физические свойства
   (плотности, влажность, пределы Аттерберга, пористость, органика, карбонатность),
   классификации по ГОСТ (`type_ground`, 9 типов) и по PLAXIS, механические параметры
   (E, c, φ) и все входы физической модели CRR. Кривая сопротивления строится физически:
   **CRR(N) = β / N^(1 − α)**, где α и β выводятся из свойств грунта (model.py). Подготовка
   сохраняет **единственный** артефакт `data/dataset`; ноутбуки `1_data_analysis/1_1…1_3` только
   читают его и выполняют разведочный анализ, **глубокий анализ параметров CRR** и стратифицированное
   разбиение train/val/test.
2. **Обучение моделей (папка 2).** Обучаются базовые модели (табличные, последовательностные,
   вероятностные) и три физически-структурированные (DPI-Flow, EVT-NeuralSSM, DPI-EVT). DPI-Flow
   использует **условный RealNVP-поток** (coupling-слои с корректной density-objective через log-det);
   DPI-EVT опирается на рекуррентный EVT-движок с калибровкой и поток не использует. Каждая модель
   сама подбирает гиперпараметры через **grid search** и пишет лучшую комбинацию в `hyperparams.json`.
   Все — по единому цензур-протоколу N_liq: разжижение наблюдается точно; отсутствие разжижения с
   плоским хвостом PPR после 500 циклов считается право-цензурированной стабилизацией; отсутствие
   разжижения без стабилизации или опыт короче 500 циклов исключается из N_liq-loss/метрик, но
   остаётся в обучении динамики PPR. Практический горизонт N_liq — 3000 циклов, после которого
   разжижение считается инженерно практически ненаступившим.
   Каждая сохраняется в `models/<имя>/` с гиперпараметрами, весами и историей обучения.
> **Тесты.** `pytest` (каталог `tests/`) проверяет: единый порог события разжижения во всех
> компонентах, цензур-протокол N_liq (три режима), наличие колонок артефакта, маску цензуры в
> метриках, численную согласованность отчётных таблиц и сквозную digitrock-консистентность
> (`liquefaction_ai.data.consistency.check_artifact_consistency`). Запуск: `pytest` из корня проекта.

3. **Оценка (папка 3).** Считается полный набор метрик (траекторные MSE/MAE/RMSE,
   AUROC/AUPRC/Brier/ECE, калибровка, покрытие интервалов), проводятся абляции,
   out-of-distribution тесты и качественный разбор кейсов. Для траекторий отдельно
   раскрываются:
   - `Traj_RMSE` — ошибка по всему наблюдаемому горизонту, включая conditioning-prefix;
   - `Traj_RMSE_continuation` и balanced/worst-state варианты — ошибка только после префикса,
     то есть собственно continuation-прогноз.

## Протокол научной интерпретации

- **Размер данных.** В лабораторной геотехнике ~1000 циклических опытов — большой датасет, а не
  малая игрушечная выборка. При этом отчётные таблицы обязаны раскрывать мощность по объектам:
  `N_CRR_test` и `N_CRR_objects`, потому что CRR-выводы могут опираться на меньшее число
  площадок/серий, чем общий risk/PPR benchmark.
- **Prefix-conditioning.** Основная постановка — прогноз продолжения после раннего PPR-префикса.
  Поэтому claims о качестве траекторного прогноза следует делать по post-prefix метрикам; full
  trajectory RMSE остаётся диагностикой реконструкции/согласованности.
- **Монотонность.** Близкая к нулю (`≤0.2%`) `Physics_Violation_Rate` у структурных моделей — это
  следствие архитектурной монотонной проекции (`cummax`/clamp) при принятом недренированном допущении
  накопления PPR, а не независимое эмпирическое доказательство физической истинности модели. Малый
  остаток (а не строгий 0) у DPI-Flow возникает из-за обучаемого траекторного residual поверх
  ODE-кривой; чисто ODE/black-box-ветви (`wo_ode`, `blackbox_raw`) дают ровно 0.
- **Насыщенные risk-метрики.** AUROC/AUPRC могут быть близки к 1 из-за сильного физического
  сигнала и prefix-conditioning. Для проверки устойчивости предусмотрены stress-протоколы:
  `stress_split(..., no_prefix=True)`, grouped/site-held-out, OOD по грунту/CSR и обучение с
  `use_observed_aux_loss=False` для режима без производных `g_obs`/`risk_proxy`.
- **P³ ranking.** P³ — заранее заданный инженерный профиль ранжирования
  `p3-prefix-conditioned-v2`, а не замена полноценной таблицы метрик. Основной траекторный
  компонент P³ предпочитает post-prefix balanced RMSE, если колонка доступна, и только затем
  откатывается к full-horizon RMSE.
- **Demo vs publication training.** `baseline_epochs` и `physics_epochs` — быстрые demo/sanity
  значения для ноутбуков. Для отчётного сравнения используйте `publication_baseline_epochs` и
  `publication_physics_epochs` либо явно зафиксированный multi-seed протокол.

## Визуализация и сохранение фигур

Все графики строятся на **matplotlib** в едином пастельном стиле публикационного уровня
(`liquefaction_ai.viz`). Построители принимают флаг `save`: при `save=True` фигура сохраняется
в `results/figs/` под именем вида `{номер_папки}_{подноутбук}_{название}` (например
`2_1_training_ft_transformer`) — в растровом `.png` (300 dpi) и векторном `.pdf` для статьи.
В ноутбуках сохранением управляет переменная `SAVE_FIGS`.

## Запуск

```bash
poetry install            # включая TDA-зависимости: ripser, kmapper, umap-learn, kaleido
poetry run jupyter lab
```

Ноутбуки выполняются по порядку: **подготовка (`data/prepare_dataset.ipynb`) → папка 1 (анализ) →
2 (обучение) → 3 (оценка) → 4 (топология)**. Каждый ноутбук самодостаточен: он сам находит корень
репозитория и добавляет `src` в путь импорта. Корневых `.py`-скриптов нет — вся логика в ноутбуках
и в библиотеке `src/liquefaction_ai/`; запуск = выполнить ноутбуки по порядку в Jupyter.

**Порядок перезапуска (полный пайплайн + reviewer-response P0–P2):**

1. `data/prepare_dataset.ipynb` — **единый ноутбук подготовки**: выбор `synthetic`|`real_objects`
   (через `LIQ_DATASET`), чтение сырых пиклов из `data/sites` (вся логика — в
   `liquefaction_ai.data.raw_loader`) либо генерация синтетики, и сохранение **единственного** артефакта
   `data/dataset` (`save_population_artifact`). Если сырья нет — используется уже сохранённый артефакт.
   Затем ноутбуки `1_data_analysis/1_1…1_3` только **читают** `data/dataset` и анализируют.
   ⚠️ С фиксом утечки префикса (`config.prefix_strict_preonset=True`): префикс обрезается строго до onset,
   иначе AUROC≈1.0 — артефакт утечки метки через вход.
2. `2_model_training/2_1 … 2_4` — бейзлайны, DPI-Flow, EVT-NeuralSSM, DPI-EVT. Каждая из трёх
   структурных моделей сама подбирает гиперпараметры **grid search** (разнообразные сетки: ёмкость,
   калибровка θ, схема интегрирования ODE, способ построения CRR) и пишет лучшую комбинацию в
   `models/<имя>/hyperparams.json`.
3. `3_evaluations/3_1 … 3_3` — лидерборд, абляции/OOD, кейсы.
4. `3_evaluations/3_4 … 3_8` — **object-CV + 95% CI** (3_4), **значимость Wilcoxon+Holm + bootstrap**
   (3_5), **абляции** (3_6), **публикационные фигуры/таблица** (3_7), **консистентность + P³-sensitivity**
   (3_8). Логика — в `liquefaction_ai.evaluation.{cross_validation,significance,ablation_study,publication}`
   и `liquefaction_ai.data.consistency`.
   Все публикационные фигуры — **на английском** (`evaluation.publication`: reliability diagram, forest-plot с object-cluster CI, Pareto-баланс onset↔trajectory, ablation-bars).

**Зависимость от TDA-библиотек.** Блоки, использующие `ripser`/`kmapper`, корректно
пропускаются при отсутствии этих библиотек (см. `liquefaction_ai.topology.tda_available`).
**Зависимость от сырых данных.** Ноутбук `4_2` дополнительно читает сырые пиклы опытов из папки
«Облако разжижения»; путь берётся из `LIQ_REAL_ROOT` или ищется рядом с репозиторием. Если папка
не найдена, `4_2` мягко пропускает анализ с понятным сообщением (без падения). Выбор датасета
(`real_objects` / `synthetic`) — через `LIQ_DATASET` либо ноутбук `data/prepare_dataset.ipynb`.

**Протокол оценки.** Основной результат — **site-held-out (grouped) оценка**:
`group_split_by_object=true` в конфиге артефакта, поэтому ни один объект не делится между
train/validation/test. Within-site стратифицированный сплит остаётся дополнительным режимом,
который можно включить явно через `group_split_by_object=false`. Тесты — `pytest` из корня проекта.
